# Technical Decisions Log

Every non-obvious decision in this codebase, why it was made, and what breaks if you change it.

---

## D1: Prisma 7 with libsql adapter for SQLite

**Decision:** Use Prisma 7 + `@prisma/adapter-libsql` instead of Prisma 5/6 with native SQLite.

**Why:** Prisma 7 was the latest version at build time. It removed `url` from the `datasource` block in `schema.prisma`, requiring `prisma.config.ts` and a driver adapter.

**Consequence:** Three extra dependencies (`prisma@7`, `@prisma/adapter-libsql`, `@libsql/client`). A `resolveDbUrl()` workaround in `src/lib/db.ts` because libsql rejects relative `file:` paths. An `as any` cast on the PrismaClient constructor.

**What breaks if changed:** If you downgrade to Prisma 5/6, you must: (1) move the URL back into `schema.prisma` datasource block, (2) delete `prisma.config.ts`, (3) remove the adapter and libsql deps from db.ts, (4) update import scripts. If you upgrade Prisma further, check the adapter API.

**Alternative considered:** Drizzle + better-sqlite3 (simpler for SQLite, no adapter dance). Not chosen because Prisma was already specified in the PRD.

---

## D2: Asset.type is a String, not an enum

**Decision:** Store `"PHOTO"` / `"VIDEO"` as a plain String column.

**Why:** SQLite has no native enum type. Prisma's `enum` declaration creates a check constraint on Postgres but is unsupported on SQLite and causes migration errors.

**Consequence:** No DB-level validation. The import script enforces the values. Queries must use `.toUpperCase()` when filtering.

**What breaks if changed:** If you add a Prisma `enum AssetType { PHOTO VIDEO }` and change the field type, the next migration will fail on SQLite. Only do this after migrating to Postgres (v0.4+).

---

## D3: Hand-built UI primitives instead of shadcn CLI

**Decision:** Write all UI components in `src/components/ui/` manually using Radix UI + class-variance-authority.

**Why:** The shadcn CLI registry was unreachable during initial build. The components use the exact same underlying libraries (Radix primitives, CVA, clsx, tailwind-merge) and follow the same API patterns.

**Consequence:** Components are functionally identical to shadcn output but weren't generated by the CLI. Running `npx shadcn add <component>` may conflict with existing files.

**What breaks if changed:** Running `npx shadcn@latest init` will try to overwrite `src/components/ui/` and may change import paths. If you want to use the shadcn CLI later, either (1) delete `src/components/ui/` first and re-add components, or (2) configure shadcn to use a different output directory.

---

## D4: Cookie-based auth with plaintext comparison

**Decision:** Auth is a single httpOnly cookie (`wg-auth=authenticated`) set after comparing the submitted password against `process.env.GUEST_PASSWORD` in plaintext.

**Why:** This is a local prototype. Real auth (Supabase Auth, NextAuth, bcrypt hashing) adds complexity with no benefit when the app runs on localhost for one person.

**Consequence:** No session expiry logic, no user identity, no password hashing. Cookie lasts 30 days.

**What breaks if changed:** If you switch to NextAuth or Supabase Auth, update: `src/proxy.ts` (cookie check logic), `src/app/api/auth/guest-password/route.ts` (replace with new auth flow), `src/app/auth/page.tsx` (login form). The `SiteSetting.guestPasswordHash` field exists in the schema for future bcrypt use.

---

## D5: Middleware excludes all /api/* from auth

**Decision:** `src/proxy.ts` does not check auth for any `/api/*` route.

**Why:** API routes serve media files (thumbnails, previews, downloads) that are referenced by authenticated pages. Adding auth checks to every API route would require passing cookies in every `<img src>` and `<video src>` tag, which browsers don't do automatically for resource loads in some cases.

**Consequence:** API routes are technically accessible without auth if someone knows the URL. Acceptable for local MVP.

**What breaks if changed:** If you add auth checks to `/api/media/*`, you must ensure all `<img>` and `<video>` tags include credentials. This is non-trivial for static resource loading.

---

## D6: Thumbnails stored as relative URLs, served through API

**Decision:** `Asset.thumbnailPath` stores a relative URL like `/generated/thumbnails/<id>.webp`. The thumbnail API route reads the file from `public/` + that path.

**Why:** Decouples the physical file location from the URL structure. The API route can serve fallback SVG placeholder if the file is missing.

**Consequence:** Every thumbnail request goes through a Node.js route handler, not static file serving. Acceptable for local use but will need CDN for production.

**What breaks if changed:** If you switch to serving from `public/` directly (static files), you lose the fallback placeholder behavior and caching headers.

---

## D7: No shared Asset type — interfaces duplicated per page

**Decision:** Each page file (`highlights`, `photos`, `videos`, `selected`, `albums/[slug]`) defines its own `Asset` interface.

**Why:** Not intentional — this is tech debt from the initial build.

**How to fix:** Extract to `src/types/asset.ts` and import everywhere. Not done yet because the shape is still stabilizing. When you add a field, update ALL page files.

---

## D8: StorageProvider interface exists but is not wired up

**Decision:** `src/lib/storage/types.ts` defines `StorageProvider` and `localStorageProvider.ts` implements it, but no code imports either file. API routes construct URLs inline.

**Why:** The abstraction was created for future cloud migration (swap local → Cloudinary/S3). It's not wired up yet because the local implementation would just add an indirection layer with no benefit.

**How to wire up:** When adding a cloud provider: (1) import the provider in API routes, (2) use `storage.getThumbnailUrl(asset)` instead of template literals, (3) swap the import to switch providers.

---

## D9: Albums derived from folder names, not user-created

**Decision:** Albums are automatically created from top-level subfolder names under `media/wedding/`. There is no UI to create, rename, or delete albums.

**Why:** The owner organizes photos in folders (Highlights, Ceremony, Reception, etc.). Automatic album creation means no manual setup.

**Consequence:** Album titles match folder names. The `Highlights` folder is special — its assets get `isHighlight=true`. Files directly in `media/wedding/` go to the "All Media" album (slug: `all-media`).

**What breaks if changed:** If you add manual album creation, the import script must be updated to not overwrite user-created albums.

---

## D10: Import marks ALL assets unavailable first, then re-marks found ones

**Decision:** `scripts/import-media.ts` runs `prisma.asset.updateMany({ data: { isAvailable: false } })` at the start of every import, then sets `isAvailable: true` for each file it finds.

**Why:** Detects files that were deleted from disk since the last import. After import, any asset still `isAvailable: false` means its file is gone.

**Consequence:** If the import crashes partway through, some assets will be incorrectly marked unavailable until the next successful import.

---

## D11: Albums page uses server-side HTTP self-fetch

**Decision:** `src/app/albums/page.tsx` is a React Server Component that fetches `http://localhost:3000/api/albums` via HTTP.

**Why:** Written this way for simplicity during initial build. Server components can't use relative URLs for fetch.

**What should change:** Replace with direct Prisma query (`import { prisma } from '@/lib/db'`). This avoids the self-fetch and the `NEXT_PUBLIC_BASE_URL` dependency.

---

## D12: Google Fonts via next/font (supersedes CSS @import)

**Decision:** Playfair Display and Inter load via `next/font/google` in `src/app/layout.tsx`.

**Why:** CSS `@import` after Tailwind caused parse errors; `next/font` avoids render-blocking requests and layout shift.

**What breaks if changed:** If you remove `next/font`, restore font variables on `<html>` / `<body>` and `.font-serif` in CSS.

---

## D15: PostgreSQL from day one (no production SQLite)

**Decision:** Production uses **Supabase or Neon Postgres** via `DATABASE_URL`. Do not deploy `file:./dev.db` or plan a SQLite → Postgres migration path.

**Why:** Owner wants cloud from the start (~10k assets); SQLite is wrong for Vercel and multi-GB libraries.

**Consequence:** C1 removes libsql adapter; migrations run on cloud DB. Local dev should use dev Postgres when possible.

**See:** `docs/DEPLOY.md`, `docs/DATABASE.md`.

---

## D16: Cloudflare R2 for media (not Cloudinary default)

**Decision:** Store originals, thumbnails, and video in **R2**, serve via **Cloudflare CDN**. Not Cloudinary at default for ~10k photos + long video.

**Why:** Cost and control at scale; import already generates thumbs with sharp/ffmpeg.

**Consequence:** Implement `r2StorageProvider`; import uploads after processing; grids use CDN URLs.

**See:** `docs/STORAGE.md`, `docs/DEPLOY.md` C2–C4.

---

## D17: Dual guest access — password + family link

**Decision:** Most guests use `GUEST_PASSWORD`. Parents use a **passwordless link** with `FAMILY_VIEW_TOKEN` (e.g. `/view/[token]`).

**Why:** Owner request; parents should not type passwords.

**Consequence:** C5 adds route + proxy exclusion + cookie; link must stay private.

**See:** `docs/AUTH.md`.

---

## D18: Owner import only in phase 1

**Decision:** No guest uploads until after cloud gallery is stable.

**Why:** Scope control for first go-live.

**Consequence:** Only `import-media` / admin reindex add assets.

---


## D13: Video poster stored in both posterPath and thumbnailPath

**Decision:** When a video poster is generated, the import script stores the same path in both `Asset.posterPath` and `Asset.thumbnailPath`.

**Why:** The thumbnail API route reads from `thumbnailPath`. If video posters were only in `posterPath`, videos would show the SVG placeholder in grids instead of the poster frame.

**Consequence:** `posterPath` is redundant for now but exists for when poster and thumbnail diverge (e.g., poster = video frame, thumbnail = cropped square).

---

## D14: .gitignore uses /media/ (anchored) not media/

**Decision:** `.gitignore` uses `/media/` with a leading slash.

**Why:** Without the leading slash, `media/` also matched `src/app/api/media/`, preventing those route files from being committed. The leading slash anchors the pattern to the project root.

**What breaks if changed:** If you remove the leading slash, the API media routes will be gitignored.
